# Architecture: Infra-Alert-Tester

## Overview
The **Infra-Alert-Tester** is a framework designed to verify the behavioral correctness of the monitoring stack (Grafana Alerts, Prometheus Queries, and Telegram Notification Templates) without requiring hardware failures or system-wide rebuilds. It treats alert rules like unit tests by injecting mock metrics to trigger specific notification scenarios.

## 1. Components

### A. Testing Metadata (Nix)
Alert rules defined in `grafana-alerts.nix` are extended with a `testScenarios` attribute. This attribute defines the mock metrics required to trigger the rule.

**Example Schema:**
```nix
{
  uid = "bcachefs_device_unhealthy";
  testScenarios = {
    "evacuating_drive" = {
      metric = "node_bcachefs_device_info";
      labels = { 
        device = "99"; 
        label = "test-hdd"; 
        state = "evacuating"; 
        uuid = "test-uuid"; 
      };
      value = 1;
    };
  };
}
```

### B. Metric Injection (Backend)
The system leverages the **Node Exporter `textfile` collector**.
- **Directory**: `/var/lib/prometheus-node-exporter/`
- **Mechanism**: Prometheus merges any `.prom` file in this directory with live system metrics. Injecting a metric with unique labels ensures the corresponding alert rule fires as intended.

### C. `alert-test` Utility (Frontend)
A CLI utility (Python or Go) installed on all nodes that manages the lifecycle of mock metrics.
- **Config**: Reads `/etc/infra-test/scenarios.json`, generated by Nix from the `testScenarios` metadata.
- **Commands**:
  - `alert-test list`: Displays all available test scenarios.
  - `alert-test trigger <scenario>`: Generates the `.prom` file to trigger the alert.
  - `alert-test clear`: Removes all mock metrics to "resolve" the alerts.

## 2. Operational Workflow

1. **Definition**: When a new alert is created, a corresponding `testScenario` is defined in Nix.
2. **Deployment**: Nix provisions the alert rule in Grafana and updates the `scenarios.json` on the nodes.
3. **Validation**: Operators can SSH into any node and run `alert-test trigger <name>`.
4. **Verification**: Confirm the Telegram notification correctly parses the HTML and populates all labels.

## 3. Implementation Roadmap (for Jules)

1. **Module Update**: 
   - Update the `monitoring` module to enable the `textfile` collector in `services.prometheus.exporters.node`.
   - Ensure `/var/lib/prometheus-node-exporter` is created with proper permissions.
2. **Package Creation**:
   - Implement the `alert-test` CLI tool in `packages/alert-test/`.
3. **Data Generation**: 
   - Use Nix to serialize the `testScenarios` into the JSON configuration file.
4. **Integration**:
   - Connect the `testScenarios` in the alert rules to the `alert-test` configuration generation.
